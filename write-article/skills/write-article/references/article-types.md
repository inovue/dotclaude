# 記事の型テンプレート

記事の型ごとの構造テンプレートと、実際の記事冒頭の例を示す。

---

## 1. 逆説型

**いつ使う:** ソース素材に「一般的にはAだと思われているが、実はBだ」という知見が含まれているとき。

### 構造テンプレート

```
## [通説を提示して否定する見出し]
（通説の紹介 → なぜ間違っているか → 根拠データ（ソース素材に含まれるもののみ使用））

## [真実を示す見出し]
（正しいアプローチの解説 → コード例）
  📊 図: 通説 vs 真実の比較

## [実践で使う場合の見出し]
（具体的な適用方法 → コード例）

## [反論への回答 — こんなケースでは通説が正しい]
（通説が正しいケースをソース素材から抽出、またはソース素材の文脈から論理的に導けるケースのみ記述 → 読者が自分の状況に当てはめて判断できるようにする）

## まとめ
```

### 冒頭の例

```markdown
# useEffectは9割のケースで不要だった — React公式が推奨する代替パターン

「副作用にはuseEffect」——Reactを学ぶと最初に教わるこの常識が、
実はパフォーマンス劣化とバグの温床になっている。

React公式ドキュメントの"You Might Not Need an Effect"が示す通り、
useEffectの多くは`useMemo`・イベントハンドラ・サーバーコンポーネントで置き換えられる。
本記事では、useEffectを使うべき残り1割のケースと、9割の代替パターンを実コードで解説する。
```

## 2. Before/After型

**いつ使う:** ソース素材に「何かを改善した」「パフォーマンスが向上した」「プロセスを効率化した」という事例が含まれており、**改善の具体的な数値（レスポンスタイム、ビルド時間、エラー率等）がソース素材に明記されている**とき。数値がない素材にこの型を適用してはならない。

### 構造テンプレート

```
## Before: [改善前の状態を具体的に]
（苦痛の描写 → 数字で現状を示す → コード例: 改善前）
  📊 図: 改善前のアーキテクチャ/フロー

## なぜ遅い（悪い）のか
（原因分析 → ボトルネックの特定）

## After: [改善後の状態を具体的に]
（改善アプローチの解説 → コード例: 改善後）
  📊 図: 改善後のアーキテクチャ/フロー

## どのくらい変わったか
（ソース素材に含まれるベンチマーク結果・数値のみ使用。ソースにない数値は絶対に捏造しない。数値の文脈（測定環境・条件）もソースに記載がある場合のみ明記する）

## トレードオフと注意点
（改善に伴う副作用・制約をソース素材から抽出。例: 速度は上がったがメモリ使用量が増えた等）

## 自分のプロジェクトに適用するには
（汎用的な手順に落とし込む）

## まとめ
```

### 冒頭の例

```markdown
# CI/CDパイプラインを45分→4分に短縮した全記録

PRを出すたびに45分待たされるCI。コーヒーを淹れに行く時間すら足りない。
チーム全体で1日あたり3時間がCIの待ち時間に消えていた。

並列化・キャッシュ戦略・不要ステップの削除で4分まで短縮した。
本記事では、ボトルネックの特定方法から具体的な設定変更まで、全手順を公開する。
```

## 3. ステップバイステップ型

**いつ使う:** ソース素材が手順・チュートリアル・セットアップガイドの場合。

### 構造テンプレート

```
## 完成形を先に見せる
（最終的に何ができるかを画面キャプチャやコードで示す）
  💻 コード: 完成形のスニペット or 動作結果

## 前提条件
（必要なツール・バージョン・知識を箇条書き）

## ステップ1: [具体的な動作を示す見出し]
  💻 コード: このステップのコード全文
  📊 図: 必要に応じてフロー図

## ステップ2: ...
## ステップ3: ...

## よくあるエラーと対処法（ソース素材にエラー情報がある場合のみ）
  📊 図: エラー判別フローチャート
  ※ ソース素材にエラー情報がない場合は、このセクションを「カスタマイズ・応用例」に置き換える

## まとめ
```

### 冒頭の例

```markdown
# Next.js 15 + Supabase + Stripeで決済付きSaaSを作る完全ガイド

この記事を最後まで読むと、認証・データベース・決済を備えたSaaSのMVPが1つ完成する。

対象読者: Next.jsでページを作った経験がある人。
SupabaseとStripeは未経験でもOK。この記事で初期設定から全て解説する。
```

## 4. 比較・選定型

**いつ使う:** ソース素材が技術比較、ツール選定、アプローチの検討の場合。

### 構造テンプレート

```
## 結論（先に言う）
（忙しい人向けに結論を1段落で → 選定マトリクス図）
  📊 図: quadrant-chart で4象限マッピング

## 比較の前提
（比較軸の定義 → 評価環境 → 何をもって「良い」とするか）

## [技術A]の特徴
  💻 コード: Aでの実装例

## [技術B]の特徴
  💻 コード: Bでの同じ処理の実装例

## 詳細比較
  📊 図: 表形式 or レーダーチャート（比較数値はソース素材または公式ドキュメントから取得したもののみ使用。独自ベンチマークは捏造禁止）

## この比較が古くなるとき
（技術比較は賞味期限が短い。この記事の比較が有効な条件・バージョンを明記し、読者が将来参照したときに陳腐化を判断できるようにする）

## どう選ぶか — 判断フローチャート
  📊 図: flowchart で判断フロー

## まとめ
```

### 冒頭の例

```markdown
# Bun vs Deno vs Node.js — 2026年のサーバーサイドJSランタイムはどれを選ぶべきか

結論から言うと、**既存プロジェクトならNode.js、新規のAPIサーバーならBun、
セキュリティ最優先ならDeno**が現時点での最適解だ。

ただしこの結論は「Webサーバー用途」に限った話で、
CLIツールやスクリプト用途では答えが変わる。本記事では用途別の選定基準を示す。
```

## 5. トラブルシュート型

**いつ使う:** ソース素材にエラー解決、ハマりどころ、踏み抜きやすい罠が含まれているとき。

### 構造テンプレート

```
## この記事で解決するエラー/問題
（エラーメッセージや症状を正確に記載 → 検索でたどり着けるように）
  💻 コード: エラーが出る状況の再現コード

## 原因
（なぜこのエラーが起きるかを図解）
  📊 図: エラーが発生するメカニズムのフロー図

## 解決策
  💻 コード: 修正後のコード（差分ハイライト）

## [2つ目の問題...]
## [3つ目の問題...]

## まとめ — ハマらないためのチェックリスト
```

### 冒頭の例

```markdown
# NextAuth v5で確実にハマる5つの罠 — 全て実体験から解説

NextAuth（Auth.js）v5への移行で丸2日を溶かした。
ドキュメント通りに書いたのに動かない。エラーメッセージは意味不明。

同じ苦しみを味わう人を減らすため、ハマったポイントと解決策を全て記録する。
記事内のコードは全てv5.0.38 + Next.js 15.1で動作確認済み。
```

## 6. 概念図解型

**いつ使う:** ソース素材が抽象的な概念・アーキテクチャ・仕組みの解説で、理解のハードルが高い場合。比喩（アナロジー）は強力な教育ツールだが、不適切な比喩は誤解を生む。比喩が自然に思いつかない概念には無理に比喩を当てはめず、図解と段階的な説明で理解を導くこと。

### 構造テンプレート

```
## 一言で言うと
（概念を1文でまとめる。自然な比喩があれば使う。なければ比喩なしで直接説明する）
  📊 図: 概念の全体像をmindmapで

## [概念の核心部分]を理解する
（比喩を使う場合: 日常の比喩で説明 → 技術的な正確さに徐々に近づける）
（比喩を使わない場合: 最もシンプルなケースから段階的に複雑さを追加する）
  📊 図: 概念の構造図

## 内部で何が起きているか
（技術的な詳細 → コード例で裏付け）
  💻 コード: 概念を体感できる最小コード
  📊 図: 内部処理のsequence-diagram

## なぜこの設計になっているか
（設計思想・歴史的経緯。ソース素材に記載がある場合のみ）

## この説明の限界（比喩を使った場合は必須）
（比喩が成立しなくなる場面・高度なユースケースを明示する。読者が比喩を「正確な理解」と混同しないようにする。例: 「レストランの比喩ではCRDやNetwork Policyの概念は表現できない。ここから先は公式ドキュメントを参照」）

## 実務でどう使うか
  💻 コード: 実践的なコード例

## まとめ
```

### 冒頭の例

```markdown
# Kubernetesを"レストランの厨房"で理解する — 概念から実運用まで

Kubernetesは「コンテナのオーケストレーションツール」と説明される。
だがこの説明で理解できた人を、私は見たことがない。

本記事では、Kubernetesの全コンポーネントを
「レストランの厨房」に例えて解説する。
Pod=料理人、Service=受付カウンター、Ingress=入口の看板。
この比喩で全体像を掴んだあと、実際のyamlと対応させて理解を固める。
```
