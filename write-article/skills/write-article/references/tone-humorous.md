# ユーモア・レトリック執筆ガイド

技術記事にユーモアを入れる目的は「面白くすること」ではなく、**読者の記憶に残り、最後まで読ませること**。笑いは手段であって目的ではない。技術的な正確さは絶対に犠牲にしない。

## 3つのユーモアパターン

使い分けではなく、1つの記事内で自然に混ぜる。

### 1. 自虐・共感型

読者が「あるある」と感じる失敗体験や苦労を、自分をオチにして語る。

```
❌ 「設定ファイルの記述を間違えていました」
✅ 「3時間にわたるデバッグの末、原因はyamlのインデントがスペース3つだったことが判明した。タブ派 vs スペース派の宗教戦争に、まさか自分が巻き込まれるとは」
```

```
❌ 「本番環境で問題が発生しました」
✅ 「金曜17時のデプロイ。"軽微な修正だから大丈夫"——この言葉を信じた過去の自分を殴りたい」
```

ポイント:
- 失敗の「規模」と「原因のしょぼさ」のギャップが笑いになる
- 読者の共感を得るには具体的なシチュエーション（時間、場所、心理状態）を入れる
- 自分を下げても技術の説明は正確に保つ

### 2. 比喩・擬人化型

技術的概念を日常のモノや人間関係に例える。

```
❌ 「CIが失敗するとマージできません」
✅ 「CIは職場で一番厳しい先輩だ。コードを出すたびに赤ペンだらけで返してくる。だが、この先輩を通過したコードは本番で問題を起こさない」
```

```
❌ 「キャッシュを適切に設定することで高速化できます」
✅ 「キャッシュは冷蔵庫の作り置きと同じだ。毎回ゼロから料理するより、日曜に仕込んでおけば平日は温めるだけで済む。ただし放置しすぎると腐る。max-ageの設定が消費期限にあたる」
```

ポイント:
- 比喩は「構造が似ているもの」を選ぶ（表面的な類似ではなく、関係性の類似）
- 比喩を導入したら、技術的な正確さに着地させる。比喩で終わらない
- 1つの比喩を記事全体で引っ張りすぎない。長くても1セクション

### 3. ツッコミ・皮肉型

公式ドキュメントの不親切さ、ツールの癖、業界の慣習への「ぼやき」。

```
❌ 「ドキュメントに記載がありませんでした」
✅ 「公式ドキュメントにはこの挙動について一切書かれていない。秘伝のタレか？」
```

```
❌ 「設定項目が多く複雑です」
✅ 「設定ファイルは47行。"シンプルなセットアップ"とREADMEに書いた人の"シンプル"の定義を聞いてみたい」
```

ポイント:
- ツッコミの矛先は「モノ・状況・慣習」に向ける。特定の個人や企業を攻撃しない
- 皮肉の直後に「とはいえ、こうすれば解決する」と建設的に繋げる
- 読者が「わかる」と感じるフラストレーションを代弁する

## ユーモアの配置ルール

- **冒頭**: ユーモアで掴む。最初の3文以内に読者を笑わせるか共感させる
- **技術解説の合間**: 密度の高い説明が続いた直後に、1文の軽い比喩やツッコミを挟んで読者を休ませる
- **まとめ**: 冒頭のユーモアを回収する（伏線回収）と記事の完成度が上がる
- **コードブロックの直前/直後**: コードの前に軽いフリ、後に「これで先輩（CI）も納得」のようなオチ

## やってはいけないこと

- 技術的に不正確な比喩を「面白いから」で押し通す
- 全文にユーモアを詰め込む（疲れる。緩急が大事）
- 内輪ネタ・世代限定ネタ（読者層が狭まる）
- 読者を見下すような笑い（「こんなことも知らないの？」）
- ソース素材にない体験談の捏造（「3日デバッグした」等を勝手に追加しない。ソースに体験談があればそれを活かす）
