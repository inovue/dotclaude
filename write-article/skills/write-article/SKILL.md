# write-article: Zenn/Qiita向け技術記事ライター

ソース素材から、**読者が3秒で構造を掴み、読了後に"完全に理解した"と感じる**技術記事を執筆する。

**入出力ルール**: 入力は何語でもOK。出力は常に日本語。

---

## コマンドルーティング

引数を確認:

- **`config`** → Config管理モード（「Config管理」セクションへ）
- **URL or ファイルパス** → 記事執筆モード（「執筆ワークフロー」へ）
- **引数なし** → ユーザーに確認

---

## Config管理

### configファイル

パス: プロジェクトルートの `.article-writer.yml`

```yaml
presets:
  - name: "zenn-tech"
    platform: "zenn"
    article_type: "step-by-step"
    visual_density: "balanced"
    output_dir: "./articles"
    default: true
```

### config項目

| キー | 説明 | 必須 |
|-----|------|-----|
| `name` | プリセット名（一意） | Yes |
| `platform` | `zenn` or `qiita` | Yes |
| `article_type` | 記事の型（後述の6型から選択） | Yes |
| `visual_density` | `light`(1〜2図) / `balanced`(3〜5図) / `rich`(6〜8図) | Yes |
| `output_dir` | 出力ディレクトリ | Yes |
| `default` | デフォルトプリセットか（`true`は1つのみ。複数ある場合は最初の1つを使用） | No |

### config操作

`/write-article config` 実行時:
1. `.article-writer.yml` の有無を確認
2. 存在する → プリセット一覧を表示、追加/編集/削除を選択
3. 存在しない → 対話形式で各項目を設定（各項目3〜5個の選択肢を番号付きで提示）

## 記事の6つの型

素材に最適な型を選ぶ。型ごとの詳細テンプレートは `${CLAUDE_SKILL_DIR}/references/article-types.md` の**該当する型のセクションだけ**を参照。

| 型 | 適する素材 | タイトル例 |
|---|---|---|
| **逆説型** | 通説を覆す知見 | 「useEffectは9割不要だった」 |
| **Before/After型** | 改善・最適化の事例 | 「デプロイを1時間→3分にした話」 |
| **ステップバイステップ型** | 手順・チュートリアル | 「Docker+Next.jsの環境構築を完全解説」 |
| **比較・選定型** | 技術選定・ツール比較 | 「Bun vs Deno vs Node.js 2026年版」 |
| **トラブルシュート型** | エラー解決・ハマりどころ | 「NextAuthでハマった5つの罠と解決法」 |
| **概念図解型** | 仕組み・アーキテクチャ解説 | 「Kubernetesを"料理のレシピ"で理解する」 |

## 執筆ワークフロー

### STEP 0: 準備

1. **ソース読み込みと保存**:
   - URL → WebFetchで取得。失敗時は「ページの内容を取得できませんでした。以下のいずれかで提供してください: (1) 内容をテキストでコピペ (2) ローカルファイルとして保存したパス」と案内
   - ローカルファイル → Readで読み込み
   - **取得したソース素材を `/tmp/article-source-[slug].txt` に保存する**（STEP 3-2の事実検証で使用）
   - WebFetchで取得したが内容が極端に短い（500文字未満）、またはHTMLタグだけで本文がない場合は、JavaScript SPAの可能性がある。ユーザーにテキストでの提供を依頼する
2. **config確認**:
   - `.article-writer.yml` がなければ、platform と article_type を聞く（最低この2つで執筆可能）

### STEP 1: ソース分析（内部処理。ただし核心が絞れない場合はユーザーに確認）

素材を分析し、以下を整理:
- 核心メッセージ（1文で言えるか？言えないなら素材が複数トピックを含んでいる。どこにフォーカスするかユーザーに確認する）
- 読者が得る具体的なベネフィット
- 図解すべきポイント（後述の判断基準で選定）
- コード例として抽出すべき箇所
- 最適な記事の型（configで指定済みならそれに従う）

**型選定の制約（素材と型の適合性チェック）:**
- **Before/After型**: ソース素材に改善の具体的な数値（レスポンスタイム、ビルド時間、エラー率等）が含まれている場合のみ選択可。数値がない場合はユーザーに「具体的な改善数値はありますか？」と確認し、提供できなければ別の型を推奨する
- **比較・選定型**: ソース素材に比較対象の具体的な特徴が含まれている場合のみ選択可。ベンチマーク数値はソース素材に含まれるもののみ使用する
- **トラブルシュート型**: ソース素材に実際のエラーメッセージ・再現手順が含まれている場合に最適

### STEP 2: タイトル＋冒頭＋構成の提案（1回の承認で進める）

**核心メッセージを1文で**提示し、続けて**タイトル3案**をセットで提示。各案には**冒頭文と構成案**を含める。ユーザーは1回の選択で記事の方向性を全て決定できる。

タイトルの鉄則:
- 具体的な数字を入れる（「5選」「3つの理由」）
- 読者のベネフィットが見える（「〜できるようになる」）
- 意外性がある（「実は〜だった」）
- 短すぎず長すぎず。内容が伝わる最短の長さを目指す

冒頭の鉄則（技術記事は「結論ファースト」）:
- 1文目で「この記事で何がわかるか」を明示
- 2〜3文目で「なぜそれが重要か」を示す
- 読者の時間を尊重する。前置きは不要

```
📝 ソース分析結果:
核心: 「○○は△△であり、□□することで解決できる」

【案1: 逆説型】
タイトル: 「〇〇は実は△△だった — 現場で気づいた3つの事実」
冒頭: 「〇〇を使っていて"なんか遅い"と感じたことはないだろうか。
原因は△△にある。本記事では〜〜を解説する。」
構成:
  ## 通説: 〇〇はこうするもの
  ## 真実: 実は△△だった → 📊 図: 比較
  ## 実践で使うには → 💻 コード
  ## こんなケースでは通説が正しい
  ## まとめ

【案2: ...】
【案3: ...】

番号で選択、またはカスタム入力:
```

### STEP 3: 執筆 → レビュー → 出力（連続実行、承認不要）

承認後、以下を一気に実行する。

#### 3-1. 執筆

**情報の補完ルール（全執筆ルールに優先）:**

技術記事はソース素材の転載ではない。読者のために情報を再構成・補完することが記事の価値。ただし、補完してよい情報と補完してはならない情報の境界を厳守する。

| 補完してよい | 補完してはならない |
|---|---|
| 用語の定義・解説（公知の事実） | 数値・ベンチマーク・実測値 |
| 前提知識の説明 | バージョン固有の挙動・仕様 |
| コードの読み方の解説 | API名・オプション名・デフォルト値 |
| 概念の構造化・分類 | 「動作確認済み」「検証済み」の主張 |
| 記事の構成・流れ | 他人の発言・引用 |
| 読者への問いかけ・動機づけ | ライセンス・利用規約に関する情報 |

「補完してはならない」情報が記事に必要な場合:
1. まずソース素材を確認する
2. ソースにない場合はWebFetchで公式ドキュメントを取得して確認する
3. 確認できない場合はその情報を記事から削除する（「未確認」「推定」として残すことも禁止）

**コード例の扱い（技術記事の心臓）:**

コード例は読者が最も求めているもの。以下のルールで配置:
- **3段落以上、コード・図・テーブルのいずれも出てこない区間を作らない**（技術記事でテキストだけの区間が長いと離脱する）
- コードは**コピペで動く状態**で提供する。省略(`...`)は原則禁止
- 長いコードは**差分形式**で示す（変更行にコメントで `// ← 追加` `// ← 変更`）
- コードの直前に「何をするコードか」を1文で書く。直後に「何が起きるか」を1文で書く

**わかりやすさの実装（抽象原則ではなく変換ルール）:**

| やりがちな書き方 | 変換後 |
|---|---|
| 「〇〇を設定します」 | 「`config.ts`の3行目に以下を追加:」 |
| 「APIを呼び出します」 | 「ブラウザで `http://localhost:3000/api/hello` を開く。`{"message":"ok"}` と表示されれば成功」 |
| 「認証を実装します」 | 「ログインボタンを押すと、Googleの認証画面に飛ぶ。その裏側の仕組みはこうなっている:」 |
| 「非同期処理を使います」 | 「洗濯機を回している間に料理ができるのと同じ。awaitは"洗濯が終わるまで待つ"という宣言」 |
| 「パフォーマンスが向上します」 | 「レスポンスが800ms→120msになる（実測値）」 |

これらは例示であり、核心は**読者の画面に何が映っているか・読者の手が何をするかを実況中継するレベルの具体性**を持たせること。抽象的な動詞（「設定する」「実装する」「活用する」）は、読者に「で、具体的に何をすればいいの？」と思わせる。

**文章のリズム:**
- 1文は自然な長さで書く。無理に短くしない。ただし1文に2つ以上の情報を詰め込まない
- 同じ概念には同じ言葉を使う（「設定」と「コンフィグ」を混在させない）
- 専門用語は初出時に1行で説明を添える。英語の技術用語は原語のまま使い、初出時に日本語訳を括弧で添える（例: Authentication（認証））
- 段落は3〜5行以内

**見出しの設計:**
- 見出しだけ読んで記事の結論がわかること
- 「〇〇について」「〇〇とは」で終わる見出しは禁止 → 「〇〇で△△する」「〇〇が必要な理由」のように内容を示す

**Mermaid図の挿入:**

図を入れるかどうかの判断基準:
- 3つ以上の要素の関係性がある → **図にする**
- 時系列の変化・手順がある → **図にする**
- 比較・分類がある → **図にする**
- 単一の事実の説明 → **図にしない**（文章で十分）
- 文章と同じことを繰り返すだけの図 → **入れない**

図を作るときの鉄則:
- 図の中のラベルは5語以内
- 読者が図を**3秒見て構造がわかる**ことがゴール
- 挿入前に、対応するリファレンスを `${CLAUDE_SKILL_DIR}/mermaid-references/` から必ず読む

主要な図の種類:

| 図 | ファイル | 使いどころ |
|---|---|---|
| Flowchart | flowchart.md | 手順・判断フロー・処理の流れ |
| Sequence | sequence-diagram.md | API通信・コンポーネント間のやりとり |
| Mindmap | mindmap.md | 概念の全体像・カテゴリ分類 |
| State | state-diagram.md | 状態遷移・ライフサイクル |
| ER Diagram | er-diagram.md | データ構造・テーブル関係 |
| Timeline | timeline.md | 時系列・経過・バージョン変遷 |
| Pie Chart | pie-chart.md | 割合・構成比 |
| XY Chart | xy-chart.md | 数値データの推移・比較 |
| Quadrant | quadrant-chart.md | 2軸比較・技術選定マトリクス |

上記以外の図種（architecture, gantt, sankey, class-diagram 等）も `mermaid-references/` に用意がある。特殊な表現が必要な場合のみ参照。

#### 3-2. アンチパターン照合（内部レビュー）

執筆完了後、出力前に以下のアンチパターンを**1つずつ照合**する。該当があれば修正してから出力する。

**構造のアンチパターン:**
- [ ] 見出しだけ抜き出して読む。記事の結論がわからなければ見出しを修正
- [ ] 各見出し間を確認。コード・図・テーブルなしに3段落以上続く区間があれば挿入
- [ ] 記事の最初の2文を読む。「何がわかるか」が明示されていなければ書き直し

**文章のアンチパターン（以下のキーワードを記事本文から検索し、前後を確認）:**
- [ ] 「解説します」「紹介します」「説明します」で終わる段落 → 具体的内容に置き換え
- [ ] 「設定する」「実装する」「活用する」「構築する」→ 直後にファイル名・コマンド・コード例がなければ具体化
- [ ] 図の直後の段落が図と同じ情報を繰り返していないか確認。重複していれば削除
- [ ] 「〜できます」「〜になります」で終わる段落 → 「具体的にどうやって？」と自問し、手順・コード・数値がなければ補足

**コードのアンチパターン:**
- [ ] ` ``` ` の直後に言語指定があるか全コードブロックを確認
- [ ] コードブロック内に `...` や `// 省略` がないか検索。あればコード全文に展開
- [ ] 各コードブロックの直前1文と直後1文を確認。「何をするか」「何が起きるか」がなければ追加

**図のアンチパターン:**
- [ ] 各Mermaid図の直前の段落を読む。図と同じ情報しかなければ段落を削除または別の情報に
- [ ] 各Mermaid図のラベルを確認。5語を超えるラベルがあれば短縮
- [ ] 各Mermaid図の構文を、対応するリファレンスの基本構文と照合
- [ ] quadrant-chart, radar-chart, requirement-diagram, xy-chart, sankey のラベルに日本語が含まれていないか確認（これらの図種は日本語ラベル非対応）

**事実検証（ソース突き合わせ）:**

STEP 0で保存した `/tmp/article-source-[slug].txt` を使って以下を検証する:

1. 記事内の全バージョン番号を抽出し、ソースファイルに同じ番号が存在するかGrepで確認する
2. 記事内の数値データ（レスポンスタイム、パーセンテージ等）を抽出し、ソースファイルに同じ数値が存在するかGrepで確認する
3. 記事内のAPI名・関数名・オプション名がソースファイルに存在するかGrepで確認する
4. 上記で**ソースに存在しない情報が見つかった場合**: WebFetchで公式ドキュメントを取得して確認するか、該当箇所を記事から削除する

#### 3-3. ファイル出力

1. プラットフォームに応じたfront matterを生成（`${CLAUDE_SKILL_DIR}/references/platform-zenn.md` または `platform-qiita.md` を参照）
2. 出力ディレクトリを確認（configの`output_dir`、なければカレント）
3. ファイル名: `YYYY-MM-DD-[slug].md`
   - slug: タイトルから主要キーワード2〜4語を英語で抽出、ハイフン結合・小文字
   - 同名ファイルが既に存在する場合は末尾に `-2`, `-3` と連番を付与
4. ファイル内容: **front matter + 記事本文のみ**（レビュー結果・メタ情報・AIコメントは一切含めない）
5. 出力パスをユーザーに通知
