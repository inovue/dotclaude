# write-article: 人間が書いた技術記事

技術記事は正確であるだけでは足りない。**読者が最後まで読み、翌日も覚えている記事**を書く。

そのために必要なのは、テンプレートではなく**声（voice）**だ。

**入出力ルール**: 入力は何語でもOK。出力は常に日本語。

---

## 声の原則 — この記事の全ルールに優先する

AIが書いた文章には声がない。均質で、整然としていて、どこを読んでも同じトーンで、同じリズムで、同じ構文で進行する。人間はそうではない。

**声とは何か:**
- 書き手が「何を面白いと思っているか」が文章に滲むこと
- 読者と同じ地平に立っていること（解説者として上から見下ろさない）
- 自分でも整理しきれていないことを、整理しながら書いている感覚

**声を実装するための具体的ルール:**

1. **一人称を使う** — 「〜される」「〜と言われている」の受動態を避け、「私は〜した」「〜だと思う」で書く。技術記事でも書き手の存在を消さない
2. **断定と留保を混ぜる** — 「これは間違いなくバグだ」と「正直、なぜ動いているのかわからない」が同じ記事に共存していい
3. **均質なトーンを壊す** — 真面目な解説の直後に1文だけ軽い感想を挟む。ずっと軽い文脈で突然真面目なことを言う
4. **語尾を同じにしない** — 「〜です」「〜です」「〜です」は死んだ文章。「〜だ」「〜である」「〜だった」「〜なのだ」「〜らしい」を混ぜる。体言止めも使う

## Config管理

### configファイル

パス: プロジェクトルートの `.article-writer.yml`

```yaml
presets:
  - name: "zenn-tech"
    platform: "zenn"
    article_type: "step-by-step"
    visual_density: "balanced"
    tone: "standard"
    output_dir: "./articles"
    default: true
```

### config項目

| キー | 説明 | 必須 |
|-----|------|-----|
| `name` | プリセット名（一意） | Yes |
| `platform` | `zenn` or `qiita` | Yes |
| `article_type` | 記事の型（後述の6型から選択） | Yes |
| `visual_density` | `light`(1〜2図) / `balanced`(3〜5図) / `rich`(6〜8図) | Yes |
| `tone` | `standard`(デフォルト) / `humorous`(ユーモア・脱線・感情のピボットを積極的に使う) | No |
| `output_dir` | 出力ディレクトリ | Yes |
| `default` | デフォルトプリセットか（`true`は1つのみ。複数ある場合は最初の1つを使用） | No |

### config操作

`/write-article config` 実行時:
1. `.article-writer.yml` の有無を確認
2. 存在する → プリセット一覧を表示、追加/編集/削除を選択
3. 存在しない → 対話形式で各項目を設定（各項目3〜5個の選択肢を番号付きで提示）

## 記事の6つの型

素材に最適な型を選ぶ。型ごとの詳細テンプレートは `${CLAUDE_SKILL_DIR}/references/article-types/` 配下の該当ファイルを参照。

| 型 | ファイル | 適する素材 |
|---|---|---|
| **逆説型** | paradox.md | 通説を覆す知見 |
| **Before/After型** | before-after.md | 改善・最適化の事例（具体的数値が必須） |
| **ステップバイステップ型** | step-by-step.md | 手順・チュートリアル |
| **比較・選定型** | comparison.md | 技術選定・ツール比較 |
| **トラブルシュート型** | troubleshoot.md | エラー解決・ハマりどころ |
| **概念図解型** | concept.md | 仕組み・アーキテクチャ解説 |

型はあくまで骨格。型に従った上で、声・リズム・脱線・具体性を自由に載せる。

## 執筆ワークフロー

> **⚠️ サブエージェント使用禁止:** 全STEPを通じて Agent ツールによるサブエージェント起動を使わないこと。サブエージェントにはメインプロセスのツール権限が引き継がれず、Read/Write/WebFetch等が拒否されて失敗する。すべてメインプロセスで直接実行する。

### STEP 0: 準備

1. **ソース読み込み**:
   - URL → WebFetchで取得し、**`/tmp/article-source-{YYYYMMDD-HHmmss}.txt` に保存する**（STEP 3-2の事実検証で使用。コンテキスト圧縮でソースが消えるリスクへの対策）。失敗時は「ページの内容を取得できませんでした。以下のいずれかで提供してください: (1) 内容をテキストでコピペ (2) ローカルファイルとして保存したパス」と案内
   - ローカルファイル → Readで読み込み。**一括読み込みせず、最初からoffset/limitで分割して読む**（例: `offset: 0, limit: 500` → `offset: 500, limit: 500` …）。または、Grepで必要箇所を特定してからピンポイントで読む。**ファイルのコピーは不要**（STEP 3-2で元ファイルに直接Grepする）
   - WebFetchで取得したが内容が極端に短い（500文字未満）、またはHTMLタグだけで本文がない場合は、JavaScript SPAの可能性がある。ユーザーにテキストでの提供を依頼する
   - **ソースのパス（保存先 or 元ファイルパス）を記録しておく**（STEP 3-2で使用）
2. **config確認**:
   - `.article-writer.yml` がなければ、platform と article_type を聞く（最低この2つで執筆可能）

### STEP 1: ソース分析（内部処理。ただし核心が絞れない場合はユーザーに確認）

素材を分析し、以下を整理:
- 核心メッセージ（1文で言えるか？言えないなら素材が複数トピックを含んでいる。どこにフォーカスするかユーザーに確認する）
- 読者が得る具体的なベネフィット
- 図解すべきポイント（後述の判断基準で選定）
- コード例として抽出すべき箇所
- 最適な記事の型（configで指定済みならそれに従う）
- **書き手はどういう立場でこの素材に触れたか** — 「自分で試した人」「調査した人」「ハマって解決した人」のどれか。声の設計に使う

**型選定の制約（素材と型の適合性チェック）:**
- **Before/After型**: ソース素材に改善の具体的な数値（レスポンスタイム、ビルド時間、エラー率等）が含まれている場合のみ選択可。数値がない場合はユーザーに「具体的な改善数値はありますか？」と確認し、提供できなければ別の型を推奨する
- **比較・選定型**: ソース素材に比較対象の具体的な特徴が含まれている場合のみ選択可。ベンチマーク数値はソース素材に含まれるもののみ使用する
- **トラブルシュート型**: ソース素材に実際のエラーメッセージ・再現手順が含まれている場合に最適

### STEP 2: タイトル＋冒頭＋構成の提案（1回の承認で進める）

**核心メッセージを1文で**提示し、続けて**タイトル3案**をセットで提示。各案には**冒頭文と構成案**を含める。

#### タイトルの鉄則

**良いタイトルは「読まずにはいられない引っかかり」を持つ。**

| 手法 | 例 | やっていること |
|-----|---|-------------|
| 通説の否定 | 「useEffectは9割不要だった」 | 読者の常識を揺さぶる |
| 具体的な数字 | 「デプロイを1時間→3分にした話」 | 成果の規模を一瞬で伝える |
| 既知フレーズの改変 | 「"設定より規約"の規約を読んだことがあるか」 | 元ネタとの差分が知的な引っかかりになる |
| 固有名詞＋意外な述語 | 「TypeScriptの型推論は壊れている」 | 有名技術＋ネガティブ動詞の落差 |
| 対比 | 「Bun vs Deno vs Node.js — 2026年の正解」 | 比較対象の並列で選定記事だと即座にわかる |

**禁止:**
- 「〇〇してみた」「〇〇を調べてみた」 — 行動報告ではなく知見を伝える
- タイトルだけで中身が全部わかるもの — 「TypeScriptでstring型をnumber型に変換する方法」は検索結果であって記事のタイトルではない

#### 冒頭の設計

**冒頭の仕事は1つだけ: 読者に「次の段落も読みたい」と思わせること。**

| 手法 | 例 | 使いどころ |
|-----|---|----------|
| **裸の宣言** | 「TypeScriptの型推論は壊れている。」 — 修飾語ゼロの断定から入る | paradox, concept |
| **場面描写** | 「金曜17時。デプロイ完了のSlack通知を確認して、PCを閉じた。5分後、アラートが鳴った。」 | troubleshoot, before-after |
| **問いかけ** | 「useEffectを使っていて"なんか遅い"と感じたことはないだろうか。原因は別のところにある。」 | paradox, concept |
| **結論ファースト** | 「結論から言うと、既存プロジェクトならNode.js、新規ならBunだ。ただしこの結論には条件がある。」 | comparison, step-by-step |

**冒頭で絶対にやらないこと:**
- 「この記事では〇〇について解説します」 — 解説するな。やれ
- 「〇〇とは、△△のことです」 — 辞書の定義から入るな。具体例から入れ
- 「近年、〇〇が注目を集めています」 — 誰も「近年」の話を読みたくない

提案フォーマット:
```
📝 核心: 「○○は△△であり、□□することで解決できる」

【案1】
タイトル: 〇〇
冒頭: （2〜3文）
構成: （見出しリスト + 図・コードの配置案）

【案2】...
【案3】...

番号で選択、またはカスタム入力:
```

### STEP 3: 執筆 → レビュー → 出力（連続実行、承認不要）

承認後、以下を一気に実行する。

#### 3-1. 執筆

**トーンの適用:**

configの `tone` が `humorous` の場合、`${CLAUDE_SKILL_DIR}/references/tone-humorous.md` を読み、そのガイドに従って文体を調整する。`standard` または未指定の場合はこのセクションの既存ルールのみで執筆する。

---

##### AIっぽさの排除 — 全執筆ルールの前提

以下のパターンが記事に1つでもあったら、人間の文章ではない。全て排除する。

| AIパターン | 人間の書き方 |
|-----------|------------|
| 「まず」「次に」「最後に」の三段構成 | 順序が必要でも「で、次にやったのが」「ここからが本題なのだが」で繋ぐ |
| 全段落が同じ長さ（3〜5行で均一） | 1行の段落と8行の段落が共存する |
| 「〜について解説します」「〜を紹介します」 | 紹介するな。いきなり具体例を置け |
| 「それでは見ていきましょう」 | 使うな。読者を案内する側に立つな |
| 感情の直接形容（「驚きました」「嬉しかったです」） | 感情は行動で描写する。「テストが全て通った。モニターを二度見した。」 |
| 同じ語尾の連続（「〜です」「〜です」「〜です」） | 語尾を毎文変える。「〜だ」「〜である」「〜だった」「〜らしい」体言止め |
| 「いかがでしたか」「参考になれば幸いです」 | 内容で語り終えていれば余計な結びは不要 |
| 全セクションが同じトーン | トーンにムラを作る。真面目→軽い→真面目→不意打ちの告白 |
| 「〇〇とは、△△のことです」で始まる定義 | 具体例から入り、定義は後から添える |
| 「近年、〇〇が注目を集めています」 | 主語のない「近年」は誰の視点でもない。自分の体験から入れ |
| 箇条書きで全てを説明する | 散文で語り、箇条書きは本当にリストが適切な場面だけ |
| 「重要なポイントは以下の3つです」 | 数を予告するな。読者は数えなくていい |

##### 文のリズム設計

**文章は音楽だ。同じビートが続くと退屈になる。**

- **短い文と長い文を交互に使う** — 「これがバグだった。」のような5文字の断定と、背景を語る3行の文を混ぜる
- **段落の長さもバラバラにする** — 1行だけの段落、5行の段落、コード例、また1行の段落。この不均一さが人間の文章
- **語尾のバリエーション** — 「〜だ」「〜である」「〜だった」「〜のだ」「〜らしい」「〜ではないか」を回す。体言止めも武器
- **句読点で呼吸を作る** — 長い文の中に読点を適切に打つ。三点リーダー「···」で「間」を作ることも許可する
- **一文一段落の強調** — 特に印象的な一文は、前後を改行して単独の段落にする

---

##### 具体性の実装 — 抽象を殺す

| 抽象的（AI的） | 具体的（人間的） |
|---|---|
| 「設定を変更します」 | 「`next.config.ts`の12行目に `experimental: { ppr: true }` を追加する」 |
| 「APIを呼び出します」 | 「ブラウザで `http://localhost:3000/api/hello` を開く。`{"message":"ok"}` と表示されれば成功」 |
| 「パフォーマンスが向上します」 | 「レスポンスが800ms→120msになる」 |
| 「大量のリクエスト」 | 「1秒あたり12,000リクエスト」 |
| 「非同期処理を使います」 | 「洗濯機を回している間に料理ができるのと同じ。awaitは"洗濯が終わるまで待つ"という宣言」 |
| 「認証を実装します」 | 「ログインボタンを押すとGoogleの認証画面に飛ぶ。その裏側の仕組みはこうなっている:」 |
| 「エラーが発生しました」 | 「`TypeError: Cannot read properties of undefined (reading 'map')` が出る」 |
| 「ドキュメントに記載がありませんでした」 | 「公式ドキュメントを端から端まで読んだ。答えは2019年のGitHub Issueの、closedされたスレッドの、最後から3番目のコメントにあった」 |

核心: **読者の画面に何が映っているか・読者の手が何をするかを実況中継するレベルの具体性**を持たせる。

---

##### 脱線の許可 — 直線的に書くな

技術記事は一直線に情報を並べると教科書になる。教科書は正確だが、退屈だ。

**許可される脱線:**
- 技術の背景にある歴史・設計思想への短い言及（「なぜこんな設計になったかというと、2015年当時は…」）
- 類似技術・代替手段への横道（「ちなみにRustだとこの問題は型レベルで防げる。が、今回はTypeScriptの話なので戻る」）
- 個人的な感想の挿入（「正直、このAPIの命名は理解に苦しむ」）

**脱線のルール:**
- 脱線は**3段落以内で本線に戻る**。戻れないなら別記事の素材
- 脱線したことを自覚する一文を入れてもいい（「話がそれた。」「本題に戻る。」）
- 脱線が本題の伏線になると最高

---

##### 読者エンゲージメント — 技法ではなく態度

「4段落ごとに問いかけを入れる」のような機械的ルールは捨てる。代わりに、以下の**態度**で書く:

- **読者と同じ場所に立つ** — 「これはおかしいと思わないだろうか」「ここで一つ疑問が出てくる」。読者が感じるであろう疑問を先回りして言語化する
- **具体で景色を見せる** — 数字は「説得」ではなく「景色を見せる」ために使う。「大規模システム」ではなく「1日200万リクエストを捌くAPIサーバー」
- **感情は行動で書く** — 「テストが通った。モニターを二度見した。」「エラーログを見た瞬間、手が止まった。」ただし**ソース素材に感情描写がある場合のみ活用し、捏造しない**
- **退屈しそうなタイミングで変化を入れる** — 技術解説が長く続いたら、短いコメント、図、コード例、小見出しで視覚的な変化を作る。ただし「4段落」のような機械的カウントはしない。退屈かどうかは文脈で判断する

---

##### まとめ・結末の設計

まとめは**記事で最も重要なセクション**だ。ここがダメなら読後感がゼロになる。

| 手法 | 説明 | 効果 |
|-----|------|------|
| **冒頭回収型** | 冒頭で提示した問題・比喩・疑問に答えて閉じる | 記事が1つの物語として完結する |
| **次のアクション型** | 読者が記事を閉じた後に何をすべきか、1ステップを提示 | 「今すぐ試せる」感覚を残す |
| **視座の引き上げ型** | 個別のトピックから一段引いて、広い文脈での意味を示す | 「この記事で扱った以上の世界がある」と示唆する |
| **余韻型** | 最後の1文で記事全体の意味をリフレームする | 読者が「そういうことか」と気づく |

**禁止:**
- 「いかがでしたか」「参考になれば幸いです」 — 虚無
- 記事の要約を繰り返すだけのまとめ — 読者はさっき読んだ
- 「今後も〇〇について発信していきます」 — 読者はあなたのフォロワーではない

---

##### 見出しの設計

- 見出しだけ読んで記事の結論がわかること
- 「〇〇について」「〇〇とは」で終わる見出し禁止 → 「〇〇で△△する」「〇〇が必要な理由」のように内容を示す
- tone=humorousの場合: 見出しにもユーモアを入れてよい

---

##### 情報の補完ルール（全執筆ルールに優先）

技術記事はソース素材の転載ではない。読者のために情報を再構成・補完することが記事の価値。ただし、補完してよい情報と補完してはならない情報の境界を厳守する。

| 補完してよい | 補完してはならない |
|---|---|
| 用語の定義・解説（公知の事実） | 数値・ベンチマーク・実測値 |
| 前提知識の説明 | バージョン固有の挙動・仕様 |
| コードの読み方の解説 | API名・オプション名・デフォルト値 |
| 概念の構造化・分類 | 「動作確認済み」「検証済み」の主張 |
| 記事の構成・流れ | 他人の発言・引用 |
| 読者への問いかけ・動機づけ | ライセンス・利用規約に関する情報 |

「補完してはならない」情報が記事に必要な場合:
1. まずソース素材を確認する
2. ソースにない場合はWebFetchで公式ドキュメントを取得して確認する
3. 確認できない場合はその情報を記事から削除する（「未確認」「推定」として残すことも禁止）

---

##### コード例の扱い

コード例は読者が最も求めているもの。以下のルールで配置:
- コードは**コピペで動く状態**で提供する。省略(`...`)は原則禁止
- 長いコードは**差分形式**で示す（変更行にコメントで `// ← 追加` `// ← 変更`）
- コードの直前に「何をするコードか」を1文で書く。直後に「何が起きるか」を1文で書く
- 技術解説が長く続いてコードが出てこない場合は、読者が退屈し始めている。コード例・図・テーブルで視覚的変化を作る

##### Mermaid図の挿入

configの `visual_density` によって図にするかどうかの判断閾値が変わる。密度が上がるほど「図にする」基準が緩くなる:

| 判断基準 | light | balanced | rich |
|---|---|---|---|
| 3つ以上の要素の関係性 | 記事の核心に直結する場合のみ図にする | **図にする** | **図にする** |
| 2つの要素の関係性 | 図にしない | 図にしない | 文章だけでは伝わりにくい場合は**図にする** |
| 時系列の変化・手順 | 5ステップ以上なら図にする | **図にする** | **図にする** |
| 比較・分類 | 文章で十分伝わるならスキップ | **図にする** | **図にする** |
| 補足的な構造（全体像、概念マップ等） | 図にしない | 理解を大きく助ける場合のみ | **積極的に図にする** |
| 単一の事実の説明 | 図にしない | 図にしない | 図にしない |
| 文章と同じことを繰り返すだけの図 | 入れない | 入れない | 入れない |

`visual_density` 未指定の場合は `balanced` として扱う。

図を作るときの鉄則:
- 図の中のラベルは5語以内
- 読者が図を**3秒見て構造がわかる**ことがゴール

#### 3-2. 最終チェック（内部レビュー）

執筆完了後、出力前に以下を確認する。

**声のチェック（最重要）:**
- [ ] 記事を音読したとき、全体が同じトーン・同じリズムになっていないか？ — ムラがなければ失敗
- [ ] 「〇〇について解説します」「それでは見ていきましょう」「いかがでしたか」が含まれていないか？
- [ ] 同じ語尾が3回以上連続していないか？
- [ ] 全段落が同じ長さ（±1行以内）に収まっていないか？ — 不均一であることが正しい
- [ ] 冒頭の1〜2文で読者の関心を引いているか？ — 定義や一般論から始まっていたら書き直す
- [ ] 書き手の「声」が聞こえるか？ — 一人称、感想、疑問、断定のいずれかが記事のどこかにあるか

**構造のチェック:**
- [ ] 見出しだけ抜き出して読む。見出しの流れだけで記事の結論がわかるか？
- [ ] まとめが冒頭と呼応しているか？ 単なる要約の繰り返しになっていないか？
- [ ] 技術解説が長く続く区間で、読者が退屈するポイントがないか？（コード・図・短いコメントで変化を作る）

**事実検証（ソース突き合わせ）:**

STEP 0で記録したソースのパス（URLソースの保存先 or ローカルファイルの元パス）を使って以下を検証する:

1. 記事内のバージョン番号・数値データ（レスポンスタイム、パーセンテージ等）をソースファイルにGrepで確認
2. 記事内のAPI名・関数名・オプション名がソースファイルに存在するかGrepで確認
3. **ソースに存在しない情報が見つかった場合**: WebFetchで公式ドキュメントを取得して確認するか、該当箇所を記事から削除する

**図の整合性チェック:**
- [ ] 特化図種を使った場合、リファレンスの基本構文と照合
- [ ] 「英語のみ」制約がある図種（XY Chart, Quadrant, Radar, Sankey, Requirement）のラベルに日本語が含まれていないか

#### 3-3. ファイル出力

1. プラットフォームに応じたfront matterを生成（`${CLAUDE_SKILL_DIR}/references/platform-zenn.md` または `platform-qiita.md` を参照）
2. 出力ディレクトリを確認（configの`output_dir`、なければカレント）
3. ファイル名: `YYYY-MM-DD-[slug].md`
   - slug: タイトルから主要キーワード2〜4語を英語で抽出、ハイフン結合・小文字
   - 同名ファイルが既に存在する場合は末尾に `-2`, `-3` と連番を付与
4. ファイル内容: **front matter + 記事本文のみ**（レビュー結果・メタ情報・AIコメントは一切含めない）
5. 出力パスをユーザーに通知
